**Related Task:** TASK-036
**Severity:** P0 — SHIP BLOCKER
**Date:** 2026-03-15

## What Happened

TASK-036 was supposed to fix the FTS5 build tag issue so that `go test ./...` and `go install` work without `-tags sqlite_fts5`. The acceptance criteria stated:

> `go test ./...` (no build tags) passes all 15 packages

The coder added `.envrc` (requires direnv) and Makefile `export CGO_CFLAGS` changes. These only work when using `make test` or when direnv is active — they do NOT fix `go test ./...` or `go install`.

**Current state:** `go test ./...` without tags still fails 12 of 18 packages.

AUDIT merged this despite the acceptance criteria not being met.

## The Correct Fix

Create `internal/storage/cgo_flags.go` with EXACTLY this content:

```go
package storage

// Enable SQLite FTS5 extension unconditionally when building with CGo.
// This eliminates the need for the -tags sqlite_fts5 build tag.

// #cgo CFLAGS: -DSQLITE_ENABLE_FTS5
import "C"
```

This makes FTS5 available whenever the storage package is compiled with CGo (which is always, since SQLite requires CGo). No build tags needed.

## Verification

After the fix, ALL of these must pass:
- `go test ./...` (no tags, no make, no direnv)
- `go build ./cmd/engram` (no tags)
- `make test`

## Impact

Without this fix, any user who does `go install github.com/TomOst-Sec/colony-project/cmd/engram@latest` gets a binary that crashes on first use.
