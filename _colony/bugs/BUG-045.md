# BUG-045: TASK-045 cgo_flags.go approach doesn't work

**Task:** TASK-045 — FTS5 Fix
**Author:** alpha-3
**Date:** 2026-03-15

## What Failed

The specified fix (creating `internal/storage/cgo_flags.go` with `#cgo CFLAGS: -DSQLITE_ENABLE_FTS5`) does NOT propagate to the `github.com/mattn/go-sqlite3` package. CGo directives only affect the package they're in. go-sqlite3 compiles SQLite within its own package, so our storage package's CGo flags don't apply to it.

## Why

Go's `#cgo` directives are per-package. They only affect CGo compilation within the package containing the directive. go-sqlite3 bundles the entire SQLite C library and compiles it within its own package. Our storage package's CGo flags have no effect on go-sqlite3's compilation.

## What Works

- `make test` works — Makefile exports `CGO_CFLAGS=-DSQLITE_ENABLE_FTS5` globally
- `go test -tags sqlite_fts5 ./...` works — build tag activates go-sqlite3's built-in FTS5 file
- `CGO_CFLAGS="-DSQLITE_ENABLE_FTS5" go test ./...` works — env var is global

## Correct Fix Options

1. **Best:** Add `GOFLAGS` to Makefile: `export GOFLAGS := -tags=sqlite_fts5` — this makes `go test ./...` inherit the tag
2. **Alternative:** Create `.envrc` with `export CGO_CFLAGS="-DSQLITE_ENABLE_FTS5"` for direnv users
3. **Alternative:** Document that `go install -tags sqlite_fts5` is required

## Recommendation

ATLAS should rewrite TASK-045 with the correct approach: use `GOFLAGS` in Makefile or accept that build tags are needed and document them properly.
