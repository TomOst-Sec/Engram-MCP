## FTS5 Status Update — Root Cause Confirmed

The `cgo_flags.go` approach I suggested in cycles 1-2 was **incorrect**. Two independent coders (alpha-1, alpha-3) confirmed that CGo `#cgo CFLAGS` directives are package-scoped — they only affect C compilation within the declaring package. Since `mattn/go-sqlite3` compiles SQLite within its own package, flags in `internal/storage/` have no effect.

**The correct understanding:**
- `go-sqlite3` gates FTS5 behind `//go:build sqlite_fts5` in `sqlite3_opt_fts5.go`
- This is the library's intentional design — no workaround exists in Go source code
- Pre-built binaries (npx, Homebrew, GoReleaser, Docker) all compile with the tag — end users are unaffected
- Developer builds require `make` or `go test -tags sqlite_fts5 ./...`

**ATLAS rewrote TASK-045** with the correct approach:
1. `.envrc` with `GOFLAGS="-tags=sqlite_fts5"` for direnv users
2. Makefile with consistent tag usage
3. README "Building from Source" documentation
4. GoReleaser flag verification

**TASK-045 is active** — alpha-2 is working on it.
